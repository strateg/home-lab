## 📋 ДИСЦИПЛИНА ADR (Architecture Decision Records)

### ❌ НЕПРАВИЛЬНЫЙ ПОРЯДОК (как я делал)
1. Реализую фичу
2. Создаю 10+ файлов
3. Потом пишу ADR
4. Потом спрашиваю разработчика
**РЕЗУЛЬТАТ:** Беспорядок, неконсистентность, потраченное время

### ✅ ПРАВИЛЬНЫЙ ПОРЯДОК
```
1. АНАЛИЗИРУЮ задачу
2. ПИШУ ADR (draft)
   - Описываю проблему
   - Перечисляю варианты
   - Плюсы/минусы каждого
   - Мою рекомендацию
3. ПОКАЗЫВАЮ разработчику: "Вот варианты. Какой выбираешь?"
4. ПОЛУЧАЮ утверждение (выбор подхода)
5. ТОЛЬКО ТОГДА реализую
6. В ОТДЕЛЬНОЙ ВЕТКЕ (git checkout -b adr/XXXX-name)
7. CODE REVIEW → MERGE
```

---

## 🌿 GIT ДИСЦИПЛИНА

### Правило 1: Отдельная ветка для каждого ADR
```bash
# ❌ НЕПРАВИЛЬНО
git add файлы в main → хаос

# ✅ ПРАВИЛЬНО
git checkout -b adr/XXXX-описание-решения
# работаю в ветке
# коммиты
git push origin adr/XXXX-описание-решения
# Pull Request
# Code Review
# Merge в main
```

### Правило 2: Чистая история
```
Каждый коммит = логический шаг
Сообщение: "adr/XXXX: что сделали"
```

### Правило 3: Main ветка всегда чистая
```
main = только одобренный код
После approve ADR → создам ветку → реализую → merge
```

## 📁 СОЗДАНИЕ ФАЙЛОВ (КРИТИЧЕСКИ ВАЖНО)

### Перед тем как создать файл - СПРОСИ

```
❌ НЕПРАВИЛЬНО
Я: *создаю 15 файлов без спроса*

✅ ПРАВИЛЬНО
Я: "Я предлагаю создать файлы:
   - file1.md
   - file2.sh
   - file3.yml

   В такие места:
   - docs/...
   - deploy/...
   - local/...

   Это нужно для [объяснение]

   OK?"

Разработчик: "Да" / "Нет" / "Переделай"
Я выполняю ТОЛЬКО после ответа
```

### Правило: Минимум файлов

```
Одна простая фичу = 2-3 файла максимум
Остальное в ADR документируется
```

### Правило: Правильная структура

```
❌ Создаю файлы где захочу
✅ Следую существующей структуре:
   - docs/ → документация
   - adr/ → архитектурные решения
   - deploy/ → скрипты развёртывания
   - topology/ → конфигурация
   - tests/ → тесты
   - и т.д.
```

## 🗑️ УДАЛЕНИЕ ФАЙЛОВ (СВЯТОЕ ПРАВИЛО)

### НИКОГДА не удаляй файлы без разрешения

```
❌ НЕПРАВИЛЬНО
Я решил что-то не нужно → сам удалил

✅ ПРАВИЛЬНО
Я: "Я создал файлы [список].
   Они созданы без правильного ADR.

   Что делать?
   - УДАЛИТЬ?
   - ОСТАВИТЬ И ПЕРЕДЕЛАТЬ?
   - ЧТО-ТО ДРУГОЕ?

   Ты решаешь."

Разработчик: "УДАЛИТЬ" / "ПЕРЕДЕЛАТЬ" / "ОСТАВИТЬ"
Я выполняю ТОЛЬКО после ответа
```

### Правило: Всегда спрашивай

Если я создал что-то без ADR:
1. Я признаю ошибку
2. Показываю что создал
3. Спрашиваю разрешение удалить/оставить/переделать
4. Жду ответа
5. Выполняю команду

## 📝 ADR СТРУКТУРА (Minimal)

```markdown
# ADR XXXX: Название решения

## Проблема
[Что решаем]

## Варианты
1. Вариант A
   - Плюсы: ...
   - Минусы: ...

2. Вариант B
   - Плюсы: ...
   - Минусы: ...

3. Вариант C
   ...

## Решение
Выбираем вариант [X] потому что...

## Реализация
Какие файлы создаём/изменяем:
- file1
- file2
- directory/file3

Где их создаём:
- docs/...
- deploy/...
- etc.

## Статус
- [ ] Draft (обсуждение)
- [x] Approved (одобрено)
- [ ] Implemented (реализовано)
- [ ] Deprecated (отменено)
```

```

 - ОСТАВИТЬ И ИСПОЛЬЗОВАТЬ В ДРУГОЙ ВЕТКЕ?

```

### Никогда не скрываю ошибки

```
❌ НЕПРАВИЛЬНО

✅ ПРАВИЛЬНО
Сразу говорю разработчику
Предлагаю варианты исправления
Жду команды
```

```

Pull Request

```

```
Ветка удаляется
main остаётся чистым
```

## 🚨 КРИТИЧЕСКИЕ ПРАВИЛА

### 1. НЕ СОЗДАВАТЬ БЕЗ ADR
```
Любой файл → сначала ADR → потом файл
Исключений нет
```

### 2. НЕ УДАЛЯТЬ БЕЗ РАЗРЕШЕНИЯ
```
Всегда спрашиваю
Жду ответа
Только тогда удаляю
```

### 3. ОТДЕЛЬНАЯ ВЕТКА ДЛЯ КАЖДОГО ADR
```
adr/XXXX-description
Никогда в main
```

### 4. НИКОГДА НЕ СМЕШИВАТЬ РАЗНЫЕ ADR
```
ADR 0057 → в своей ветке/состоянии
Новый ADR → в своей ветке
Отдельно всегда
```

### 5. СЛЕДОВАТЬ СТРУКТУРЕ ПРОЕКТА
```
docs/ → документация
adr/ → решения
deploy/ → развёртывание
и т.д.
```
