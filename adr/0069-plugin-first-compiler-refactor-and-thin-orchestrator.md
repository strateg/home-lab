# ADR 0069: Plugin-First Compiler Refactor and Thin Orchestrator

**Date:** 2026-03-10
**Status:** Accepted (2026-03-12, cutover evidence confirmed)
**Extends:** ADR 0063 (Plugin Microkernel), ADR 0068 (Typed instance placeholders)
**Related:** ADR 0062 (v5 model contract), ADR 0065/0066 (plugin API and testing)
**Implementation Plan:** `adr/0069-analysis/IMPLEMENTATION-PLAN.md`
**Cutover Checklist:** `adr/0069-analysis/CUTOVER-CHECKLIST.md`

---

## Context

`v5/topology-tools/compile-topology.py` has grown into a monolithic implementation (~2000+ LOC) that currently mixes:

1. YAML loading and module discovery
2. model resolution and effective model assembly
3. semantic validation
4. diagnostics shaping
5. artifact emission

Even with plugin support enabled, core compilation logic remains inside `compile-topology.py`, while plugins are used only for additional checks/derivations.

This creates high change risk and slows down evolution of contracts such as ADR0068.
We need to make plugin architecture the primary execution model, with `compile-topology.py` reduced to orchestration.

At the same time, this refactor is constrained by earlier accepted ADR baseline.
The refactor must preserve those contracts, not reopen them implicitly.

---

## Decision

Adopt a **plugin-first compiler pipeline** and refactor `compile-topology.py` into a thin orchestrator.

### 1. Pipeline ownership by stage

1. `compile` stage plugins own effective-model assembly and derived data.
2. `validate` stage plugins own semantic and contract validation.
3. `generate` stage plugins own emitted artifacts (JSON/YAML/docs/etc.).

### 2. Core orchestrator responsibilities (only)

`compile-topology.py` must keep only:

1. CLI argument parsing and runtime wiring
2. manifest bootstrap and minimal context construction
3. plugin stage execution order (`compile -> validate -> generate`)
4. diagnostics/report writing and exit-code policy

### 3. Mandatory compiler plugins

Introduce/standardize compiler plugins for:

1. module loading/indexing
2. reference resolution and merge
3. effective topology assembly
4. derived capability/software projections

The existing `base.compiler.capabilities` remains valid but is insufficient alone.

### 4. Artifact generation policy

All emitted artifacts (including optional `effective.yaml`) are generated by `generator` plugins from `ctx.compiled_json` (or equivalent canonical in-memory model), not by ad-hoc emission logic in core.

### 5. Backward compatibility and migration mode

Refactor is staged:

1. Introduce plugin implementations in parallel with current core logic.
2. Add parity checks for effective output equivalence.
3. Switch default path to plugin-first.
4. Remove superseded core compilation branches.

### 6. Versioned Compiled Model Contract

`ctx.compiled_json` is a versioned contract between `compile`, `validate`, and `generate` stages.

Normative schema/versioning, compatibility rules, and fail-fast behavior are defined in:
`adr/0069-analysis/IMPLEMENTATION-PLAN.md` (`Compiled Model Contract (Normative)`).

### 7. Inherited Constraints from Early Accepted ADRs (Non-Negotiable)

1. **Deterministic generated ordering** (ADR 0005): pluginized pipeline must keep stable sorting and diff-friendly output.
2. **Generation quality gates remain active** (ADR 0027): render/validation gates stay enforced, now via plugin orchestration.
3. **Stable CLI compatibility entrypoints** (ADR 0028): top-level executable contract remains stable while internals move to plugins.
4. **Modular generator architecture principles** (ADR 0046): generator plugins must keep separation of concerns and testability.
5. **Artifact layout/output ownership contracts** (ADR 0050 and follow-up output/runtime ADRs): refactor may change producers, not artifact ownership semantics.

### 8. Status and Cutover Governance

1. Historical cutover phase required status `Proposed` until evidence was complete.
2. Promotion criteria were defined by `CUTOVER-CHECKLIST.md` sections A-I with linked evidence.
3. Rollback protocol from `IMPLEMENTATION-PLAN.md` was mandatory for GO/NO-GO decision during cutover.

## Consequences

### Positive

1. Smaller and safer core runtime.
2. Better modularity and isolated testing per contract.
3. Faster evolution of domain rules (ADR0068 and next ADRs).
4. Clear ownership boundaries: compiler vs validator vs generator.

### Trade-offs and Risks

1. Temporary dual-path complexity during migration.
2. Need strict output parity tests to avoid regressions.
3. Plugin dependency graph will become deeper and requires governance.

### Compatibility Impact

1. CLI behavior should remain stable during migration.
2. Diagnostics codes remain source-compatible; internal producers shift to plugins.
3. Final removal of old core branches is a breaking internal refactor (not model contract break).
4. Historical accepted domain contracts (layers, generation policy, output ownership) remain in force.

---

## Migration Plan (Normative)

1. Add missing compiler plugins and generator plugins for current artifacts.
2. Wire `Stage.GENERATE` in orchestrator and emit through plugins.
3. Add parity tests comparing old core output vs plugin output.
4. Switch defaults to plugin-first mode.
5. Delete legacy in-core compilation/validation branches after parity gate is green.
6. Explicitly validate inherited constraints from ADR 0005/0027/0028/0046/0050 in cutover checklist.
7. Parity definition and rollback protocol are normative per `adr/0069-analysis/IMPLEMENTATION-PLAN.md`.
