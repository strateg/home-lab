## Архитектура v5 (из ADR 0062)

### Канонический контракт модели

```
Instance.object_ref → Object.class_ref → Class

Эффективное слияние:
Class.defaults → Object.defaults → Instance.overrides
```

**Жесткие правила:**
- Instance не может нарушать инварианты Class
- Object должен удовлетворять required_capabilities Class
- Instance не может запрашивать неподдерживаемые capabilities

### Упрощенная модель capabilities

**3 уровня:**
1. **Capability catalog** - канонические ID (плоские, стабильные)
2. **Class contract** - `required_capabilities`, `optional_capabilities`, `capability_packs`
3. **Object binding** - `enabled_capabilities`, опционально `enabled_packs`

**Правила:**
- Без глубоких деревьев наследования
- Vendor-only capabilities используют префикс `vendor.*`
- Продвижение object-local → class pack при переиспользовании 2+ раза

### Компиляция: YAML → JSON

**5 фиксированных стадий:**
1. `load` - загрузка YAML с разрешением include
2. `normalize` - канонизация структуры
3. `resolve` - проверка ID/ссылок + связей class-object-instance
4. `validate` - JSON schema + семантические проверки
5. `emit` - канонический JSON + диагностика

**Человеки:** редактируют YAML
**Машины:** потребляют JSON

### Dual-track структура

**Обязательное разделение:**
```
v4/                      # Legacy lane (заморожена)
  topology/
  topology-tools/
  tests/
v5/                      # Новая lane (активная)
  topology/
    class-modules/
    object-modules/
    instances/home-lab/
    model.lock.yaml
    profile-map.yaml
  topology-tools/
  tests/
v4-generated/
v4-build/
v4-dist/
v5-generated/
v5-build/
v5-dist/
```

## План миграции: 8 фаз

### Фаза 0: Заморозка и разделение рабочего пространства
**Статус:** ❌ Не начата
**Трудозатраты:** 2-3 дня
**Критичность:** **БЛОКИРУЮЩАЯ**

**Действия:**
- Создать v4/ и v5/ директории
- Переименовать артефакты в v4-* / v5-*
- Разделить CI на lanes
- Обновить пути в скриптах

**Критерии выхода:**
- Каждый PR явно нацелен на v4 или v5
- Нет новых фич в v4
- Нет новых тестов вне v4/tests или v5/tests
- Pipeline не пишут в неверсионированные корни

### Фаза 1: Инвентаризация и маппинг
**Статус:** ❌ Заблокирована Фазой 0
**Трудозатраты:** 1-2 недели

**Действия:**
- Построить таблицу маппинга v4→v5
- Классифицировать нерешенные сущности
- Выявить разрывы в capabilities

**Критерии выхода:**
- 100% активных сущностей имеют запланированные class_ref и object_ref

### Фаза 2: Покрытие модулей классов
**Статус:** ❌ Заблокирована Фазой 1
**Трудозатраты:** 2-3 недели

**Действия:**
- Определить контракты классов
- Создать capability packs
- Проверить coverage capability checker

**Критерии выхода:**
- Все целевые объекты ссылаются на существующие классы

### Фаза 3: Покрытие модулей объектов
**Статус:** ❌ Заблокирована Фазой 2
**Трудозатраты:** 2-3 недели

**Действия:**
- Определить контракты объектов
- Проверить совместимость class/object

**Критерии выхода:**
- Все целевые instances разрешаются в валидные объекты

### Фаза 4: Миграция данных топологии
**Статус:** ⚠️ Частично (скрипт миграции)
**Трудозатраты:** 1-2 недели

**Действия:**
- Добавить class_ref и object_ref для мигрированных сущностей
- Исключить legacy-only поля из новых файлов

**Критерии выхода:**
- Компиляция v5 проходит без неразрешенных ссылок

### Фаза 5: Операционализация lock и profile
**Статус:** ❌ Заблокирована Фазой 4
**Трудозатраты:** 1 неделя

**Действия:**
- Финализировать model.lock.yaml
- Финализировать profile-map.yaml (production/modeled/test-real)
- Включить strict mode в CI

**Критерии выхода:**
- Все три профиля компилируются и валидируются

### Фаза 6: Паритет генерации и валидации
**Статус:** ❌ Заблокирована Фазой 5
**Трудозатраты:** 2-4 недели

**Действия:**
- Направить генерацию v5 из канонического JSON
- Сравнить ключевые выходы v4 vs v5
- Закрыть разрывы или задокументировать различия

**Критерии выхода:**
- Чеклист паритета одобрен для production-critical артефактов

### Фаза 7: Миграция на plugin microkernel
**Статус:** ❌ Заблокирована Фазой 6 + ADR 0063
**Трудозатраты:** 3-4 недели

**Действия:**
- Внедрить plugin manifest и registry
- Мигрировать генераторы → валидаторы → хуки компилятора
- Удалить hardcoded dispatch

**Критерии выхода:**
- Подключение нового модуля не требует изменений core
- Правила зависимостей/порядка плагинов проверяются в CI

### Фаза 8: Переход и подготовка к удалению legacy
**Статус:** ❌ Заблокирована Фазой 7
**Трудозатраты:** 2 недели

**Действия:**
- Переключить основной CI на v5
- Сохранить v4 в режиме поддержки
- Подготовить план удаления legacy полей

**Критерии выхода:**
- v5 lane - дефолт для production workflow
- Процедура rollback задокументирована и протестирована

## Анализ рисков

### Критические риски (3)

1. **CR-1: Фаза 0 не выполнена** - БЛОКИРУЕТ ВСЕ
   - **Вероятность:** ВЫСОКАЯ (факт)
   - **Воздействие:** КРИТИЧЕСКОЕ
   - **Митигация:** Выполнить немедленно

2. **CR-2: Регрессия v5 в production после перехода**
   - **Вероятность:** СРЕДНЯЯ
   - **Воздействие:** КРИТИЧЕСКОЕ
   - **Митигация:** Постепенный переход, extensive parity testing, rollback готовность

3. **CR-3: Взрыв модели capabilities**
   - **Вероятность:** СРЕДНЯЯ
   - **Воздействие:** ВЫСОКОЕ
   - **Митигация:** Строгие правила продвижения, review board, 2+ правило переиспользования

### Высокие риски (8)

- Неполный v4→v5 маппинг сущностей
- Классы/объекты не покрывают все use cases
- Потеря/повреждение данных скриптом миграции
- Компилятор неполный
- Рефакторинг генераторов ломает v4
- Баги логики подстановки профилей
- Дрейф model lock
- Недооценка timeline

### Средние риски (12)

- Path guards в CI не включены
- Путаница команды о том, какую lane использовать
- Критерии паритета слишком строгие
- ADR 0063 не принят
- Баги capability checker
- Неполная deprecation legacy полей
- Несоответствие object-class
- Недостаточное тестовое покрытие
- Документация отстает
- Утечки vendor-specific логики
- Процедура rollback не протестирована
- Нестабильность plugin API

### Низкие риски (7)

- Рост размера репозитория
- Несогласованность именования классов/объектов
- Увеличение времени CI
- Разрастание example файлов
- Путаница в semantic versioning
- Ограничения diff tool
