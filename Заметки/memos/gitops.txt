  Что чем управлять

  Terraform/OpenTofu — для того, что должно быть декларативным:

  /ip address
  /ip pool
  /ip dhcp-server
  /ip dns
  /ip firewall filter
  /ip firewall nat
  /interface wireguard
  /interface vlan
  /routing

  Есть актуальный community provider terraform-routeros/routeros; он работает через RouterOS API/REST API.

  Ansible — для процедурных задач:

  первичная настройка доступа
  создание пользователя automation
  включение api-ssl / ssh
  экспорт текущей конфигурации
  проверки перед apply
  аварийный rollback
  обновление RouterOS

  У Ansible есть коллекция community.routeros с модулями API, facts, command и modify.

  Минимальный flow изменений
  git checkout -b change/firewall-rule

  ansible-playbook ansible/backup.yml

  cd terraform
  tofu plan
  tofu apply

  ansible-playbook ../ansible/precheck.yml

  git add .
  git commit -m "Add firewall rule for WinBox from LAN only"
  Важное правило

  Перед каждым изменением делай:

  /export show-sensitive file=before-change
  /system backup save name=before-change

  .rsc export — человекочитаемый, его можно diff-ить в Git.
  .backup — аварийный бинарный снимок, лучше только для того же устройства.

  Пример Terraform provider
  terraform {
  required_providers {
  routeros = {
  source  = "terraform-routeros/routeros"
  version = "~> 1.99"
  }
  }
  }

  provider "routeros" {
  hosturl  = "https://192.168.88.1"
  username = var.routeros_user
  password = var.routeros_password
  insecure = true
  }
  Пример идеи firewall rule
  resource "routeros_ip_firewall_filter" "allow_winbox_lan" {

  protocol    = "tcp"
  dst_port    = "8291"

  Rollback-стратегия

  Самый безопасный вариант для домашней сети:

  Перед apply сохранить export и backup.
  Настроить MikroTik Safe Mode вручную в WinBox/SSH перед рискованным изменением.
  Применить Terraform.
  Проверить доступ.
  Если потерял доступ — Safe Mode сам откатит сессию.
  Если доступ есть, но логика сломана — импортировать прошлый .rsc или восстановить .backup.
  Мой совет для твоего случая

  Начни не с полной миграции всего MikroTik в Terraform, а так:

  Этап 1: Ansible backup + Git diff
  Этап 2: Terraform только firewall address-list / NAT / WireGuard
  Этап 3: Terraform DHCP/DNS/VLAN
  Этап 4: полный rebuild router from config

  Не пытайся сразу импортировать весь живой MikroTik в Terraform. Лучше постепенно переносить управляемые части и помечать их комментариями

Не уверен что нудно так усложнять систему. Возможно ли упростить систему

● Согласен, давай упростим. Вот три уровня сложности:

  Вариант 1: Минимальный (Git = State)

  Git commit SHA = идентификатор состояния

  topology изменение → git commit → terraform apply
  rollback = git revert → terraform apply

  Что храним:
  - terraform.tfstate — в .gitignore, локально или remote backend
  - backups/rtr-mikrotik-chateau.rsc — в git (export перед apply)

  Плюсы: Почти нет нового кода
  Минусы: Нет автоматического rollback, ручной процесс

  ---
  Вариант 2: Средний (Git + pre/post hooks)

  Git commit → pre-apply backup → terraform apply → post-apply check
                                        ↓ fail
                                   restore backup

  Что добавляем:
  - task apply:mikrotik — делает backup, apply, check
  - backups/ директория в git с .rsc файлами

  Плюсы: Автоматический backup/restore, минимум нового кода
  Минусы: Нет истории состояний отдельно от git

  ---
  Вариант 3: Текущее предложение (ADR 0105)

  Полная система с отдельными state commits, snapshots, history.
