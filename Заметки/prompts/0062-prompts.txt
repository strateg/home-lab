  2. Критический анализ

  2.1 Избыточная сложность (Over-Engineering)

  Проблема: Архитектура спроектирована для масштабов enterprise, но применяется к home-lab с 3 устройствами.

  Введённые концепции:
  Class → Object → Instance        (3 уровня абстракции)
  topology-core / topology-base    (2+ репозитория)
  capability catalog + packs       (система capabilities)
  model.lock                       (версионирование зависимостей)
  profile overlays                 (3 типа профилей)
  5-stage compiler pipeline        (compile chain)
  E1xxx-I9xxx error catalog        (100+ кодов ошибок)

  Вопрос: Оправдывает ли добавление GL.iNet или Ubiquiti такую сложность?

  Несогласованность topology-core vs topology-base — критическая путаница.

  2.3 Преждевременное разделение репозиториев

  ADR 0059 предлагает:

  Repository B: home-lab-topology

  ADR 0061 предлагает:

  Instance repos: many (home-lab, office-lab, etc.)

  Проблемы:
  - Нет реальных внешних потребителей для topology-core
  - Overhead на поддержку версионирования между репозиториями
  - Текущий проект — единственный пользователь

  2.4 Capability Model — чрезмерная формализация

  ADR 0059 вводит трёхуровневую модель:
  1. Capability Catalog (class-module)
  2. Class Capability Contract (required/optional/packs)
  3. Object Capability Binding (enabled + vendor.*)

  Критика:
  - capability_packs и promotion rules — enterprise patterns
  - Для 3 устройств достаточно простого enum
  - Vendor namespace (vendor.*) добавляет cognitive load без пользы

  2.5 Compiler Pipeline — решение без проблемы

  ADR 0060 предлагает 5-stage pipeline:
  load → normalize → resolve → validate → emit

  Вопросы:
  - Текущий validate-topology.py работает
  - JSON как canonical artifact — зачем, если YAML достаточен?
  - AI repair loops — где реальный use-case?

  2.6 Профили (prod/modeled/test-real)

  ADR 0061 вводит:
  production:   canonical production behavior
  modeled:      virtual substitutions (CHR вместо hardware)
  test-real:    test behavior on real devices

  Критика:
  - Home-lab = production (нет staging/prod separation)
  - CHR substitution — edge case, не требующий формализации
  - Три профиля умножают complexity × 3

  5. Гармонизация: единый ADR

  Предлагаемая структура ADR 0062:

  # ADR 0062: Device Module Architecture

  **Status:** Proposed
  **Supersedes:** ADR 0058, ADR 0059, ADR 0060, ADR 0061

  ## Context
  Consolidates device abstraction proposals into minimal viable architecture.

  ## Decision

  ### 1. Device Model (simplified)
  - device_class: abstract role (router, hypervisor, sbc)
  - implementation: module reference (mikrotik, proxmox, orangepi5)
  - Module provides: generator, validator, templates

  ### 2. Module Structure
  topology-tools/
  └── modules/
      ├── mikrotik/
      │   ├── __init__.py
      │   ├── generator.py
      │   └── validator.py
      ├── proxmox/
      └── orangepi5/

  ### 3. Validation Enhancements
  - --json-report flag for machine-readable diagnostics
  - Error codes: E1xxx (parse), E2xxx (reference), E3xxx (semantic)
  - No separate compiler; extend validate-topology.py

  ### 4. Deferred Decisions
  - Repository split (criteria: 3+ external users)
  - Capability packs (criteria: 5+ device types)
  - Profile system (criteria: multi-environment deployments)

  ## Migration
  Phase 1: Create modules/ directory structure
  Phase 2: Move vendor-specific code to modules
  Phase 3: Add --json-report to validator

  ## Consequences
  + Simpler than 4-ADR approach
  + Incremental complexity growth
  - Less formal capability contracts (acceptable for current scale)

  2.1 Терминологическая консистентность

  Решение: Единая терминология:

  2.3 Capability model — переусложнён

  ADR 0059 вводит:
  Capability Catalog → Class Contract → Object Binding → vendor.*

  Проблема: 4 уровня для ~10 capabilities.

  Упрощение:

  # topology-tools/core/schemas/capabilities.yaml
  capabilities:
    routing:
      description: "L3 packet forwarding"
      classes: [router]
    vm_hosting:
      description: "Virtual machine lifecycle"
      classes: [hypervisor]
    container_hosting:
      description: "Container lifecycle"
      classes: [hypervisor, sbc, router]  # MikroTik containers

  # В Object module:
  # topology-tools/modules/mikrotik/__init__.py
  MODULE = {
      "classes": ["router"],
      "capabilities": ["routing", "firewall", "dhcp", "nat", "container_hosting"],
  }

  Правило: vendor-specific capabilities только если нет generic equivalent.

  2.4 Profile system — нужен, но проще

  ADR 0061 вводит 3 профиля: production, modeled, test-real.

  Упрощение до 2:

  ├─────────┼─────────────────────────────────────────────┤

  ├─────────┼─────────────────────────────────────────────┤

    rtr-home-main:

  2.5 model.lock — отложить

  Текущая реальность: один репозиторий, один потребитель.

  2.6 Compiler vs Validator — уточнить границу

  ADR 0060 вводит compile-topology.py, но validate-topology.py уже существует.

  Решение: Compiler = Validator + emit JSON

  validate-topology.py  →  переименовать в compile-topology.py

  3. validate    (schema + semantic)    ← существует в validators/

  def compile(topology_path, profile=None, output_dir="generated/"):
      topology = load(topology_path)
      if profile:
          topology = apply_profile(topology, profile)
      resolved = resolve(topology)
      diagnostics = validate(resolved)
      emit_json(resolved, f"{output_dir}/effective-topology.json")
      emit_diagnostics(diagnostics, output_dir)
      return diagnostics

  Phase 4: Module registry

  # topology-tools/modules/__init__.py
  from importlib import import_module

  MODULES = {}

  def register(module_path):
      mod = import_module(module_path)
      MODULES[mod.MODULE["id"]] = mod.MODULE

  def get_module(object_ref):
      for mod in MODULES.values():
          if object_ref in mod["objects"]:
              return mod
      return None

  # Auto-discover
  for name in ["mikrotik", "proxmox", "orangepi5"]:
      register(f"modules.{name}")

  5. Validator Architecture

  5.1 Base validator protocol

  # topology-core/engine/validator.py
  from abc import ABC, abstractmethod
  from dataclasses import dataclass
  from typing import List, Literal
  from pathlib import Path

  @dataclass
  class Diagnostic:
      code: str                              # E2001, W3002
      severity: Literal["error", "warning", "info"]
      message: str
      path: str                              # JSONPath: $.devices[0].wan_interface
      source: dict | None = None             # {file, line, column}
      hint: str | None = None
      autofix: dict | None = None            # {action, value}

  class YamlValidator(ABC):
      """Validates source YAML before compilation."""

      @abstractmethod
      def validate(self, yaml_data: dict, context: "ValidationContext") -> List[Diagnostic]:
          pass

  class JsonValidator(ABC):
      """Validates compiled JSON after compilation."""

      @abstractmethod
      def validate(self, json_data: dict, context: "ValidationContext") -> List[Diagnostic]:
          pass

  @dataclass
  class ValidationContext:
      project_path: Path
      topology: dict
      merged_schema: dict
      loaded_modules: dict                   # module_id -> module_manifest
      profile: str | None

  5.2 Class validator example

  # topology-core/classes/router/validators/yaml/router_refs.py
  from topology_core.engine.validator import YamlValidator, Diagnostic

  class RouterRefsValidator(YamlValidator):
      """Validates router-class devices in YAML source."""

      def validate(self, yaml_data: dict, context) -> list[Diagnostic]:
          diagnostics = []

          for i, device in enumerate(yaml_data.get("devices", [])):
              if device.get("class_ref") != "router":
                  continue

              # Check wan_interface references valid interface
              wan = device.get("wan_interface")
              if wan:
                  interfaces = device.get("interfaces", [])
                  interface_ids = [iface["id"] for iface in interfaces]
                  if wan not in interface_ids:
                      diagnostics.append(Diagnostic(
                          code="E2101",
                          severity="error",
                          message=f"wan_interface '{wan}' not found in device interfaces",
                          path=f"$.devices[{i}].wan_interface",
                          hint=f"Available interfaces: {interface_ids}"
                      ))

          return diagnostics

  5.3 Object validator example

  # topology-modules/mikrotik/validators/json/mikrotik_contract.py
  from topology_core.engine.validator import JsonValidator, Diagnostic

  class MikrotikContractValidator(JsonValidator):
      """Validates compiled MikroTik devices in JSON."""

      def validate(self, json_data: dict, context) -> list[Diagnostic]:
          diagnostics = []

          for i, device in enumerate(json_data.get("devices", [])):
              obj_ref = device.get("object_ref", "")
              if not obj_ref.startswith("mikrotik_"):
                  continue

              # Validate container capability requirements
              if "container_hosting" in device.get("enabled_capabilities", []):
                  ros_version = device.get("routeros_version", "7.0")
                  major, minor = map(int, ros_version.split("."))
                  if major < 7 or (major == 7 and minor < 4):
                      diagnostics.append(Diagnostic(
                          code="E3201",
                          severity="error",
                          message="Container hosting requires RouterOS >= 7.4",
                          path=f"$.devices[{i}].routeros_version",
                          hint=f"Current version: {ros_version}"
                      ))

              # Validate object-specific constraints
              obj_def = context.loaded_modules["mikrotik"]["objects"].get(obj_ref)
              if obj_def and obj_def.get("virtual"):
                  if device.get("lte_config"):
                      diagnostics.append(Diagnostic(
                          code="W3202",
                          severity="warning",
                          message="LTE config ignored on virtual CHR object",
                          path=f"$.devices[{i}].lte_config"
                      ))

          return diagnostics

  6. Generator Architecture

  6.1 Base generator protocol

  # topology-core/engine/generator.py
  from abc import ABC, abstractmethod
  from dataclasses import dataclass
  from pathlib import Path

  @dataclass
  class GeneratorContext:
      topology: dict                         # effective-topology.json
      output_dir: Path
      module_path: Path                      # Path to module directory
      templates_dir: Path                    # module_path / "generators" / type / "templates"
      project_config: dict                   # project.yaml overrides

  class ArtifactGenerator(ABC):
      """Base for all generators."""

      @abstractmethod
      def get_type(self) -> str:
          """Generator type: terraform, bootstrap, ansible, docs"""
          pass

      @abstractmethod
      def get_instances(self, topology: dict) -> list[str]:
          """Return instance IDs this generator handles."""
          pass

      @abstractmethod
      def generate(self, context: GeneratorContext) -> list[Path]:
          """Generate artifacts, return list of created files."""
          pass

  6.2 Object generator example

  # topology-modules/mikrotik/generators/terraform/generator.py
  from topology_core.engine.generator import ArtifactGenerator, GeneratorContext
  from jinja2 import Environment, FileSystemLoader
  from pathlib import Path

  class MikrotikTerraformGenerator(ArtifactGenerator):

      def get_type(self) -> str:
          return "terraform"

      def get_instances(self, topology: dict) -> list[str]:
          """Find all mikrotik instances in topology."""
          return [
              d["id"] for d in topology.get("devices", [])
              if d.get("object_ref", "").startswith("mikrotik_")
          ]

      def generate(self, context: GeneratorContext) -> list[Path]:
          env = Environment(loader=FileSystemLoader(context.templates_dir))
          created_files = []

          # Extract MikroTik-specific data
          data = self._extract_data(context.topology)

          # Generate terraform files
          templates = ["provider.tf.j2", "interfaces.tf.j2", "firewall.tf.j2",
                       "dhcp.tf.j2", "vlans.tf.j2"]

          output_path = context.output_dir / "terraform" / "mikrotik"
          output_path.mkdir(parents=True, exist_ok=True)

          for tpl_name in templates:
              template = env.get_template(tpl_name)
              output_file = output_path / tpl_name.replace(".j2", "")
              output_file.write_text(template.render(**data))
              created_files.append(output_file)

          return created_files

      def _extract_data(self, topology: dict) -> dict:
          """Extract MikroTik-relevant data from compiled topology."""
          mikrotik_devices = [
              d for d in topology["devices"]
              if d.get("object_ref", "").startswith("mikrotik_")
          ]

          return {
              "devices": mikrotik_devices,
              "networks": topology.get("networks", []),
              "firewall_rules": self._extract_firewall(topology),
              "vlans": self._extract_vlans(topology),
              # ...
          }
